Verification Methodology

How the verification system works and what it checks

Version: 1.0 Last Updated: 2026-02-26 Audience: Non-technical executives, stakeholders, auditors

Overview

A Proof Certificate is a structured document that accompanies any data-driven deliverable. Its purpose is to answer one question: How do we know the data is real?

When we produce a report ‚Äî whether it classifies customer service issues, measures delivery failure rates, or tracks operational trends ‚Äî the Proof Certificate provides an auditable chain of evidence from every data point back to its original source. It is the difference between "trust us" and "here is the proof."

This document explains the four pillars of our verification methodology: Source Verification, Classification Verification, Integrity Scoring, and the limitations of the system.

1. Source Verification

Source Verification answers the question: Does every data point trace back to a real, identifiable source?

How It Works

Every row in a certified dataset must satisfy two requirements:

Evidence Quote. Each row contains a direct text excerpt from the original source material. This is not a summary, not a paraphrase, and not a generated description. It is the actual text ‚Äî copied verbatim ‚Äî from the conversation, email, ticket, or record that the data point was extracted from. If a row claims a customer reported a damaged product, the evidence quote shows the customer's own words describing that damage.

Source System ID. Each row includes a unique identifier that can be looked up in the original system. For example, a Front conversation ID (e.g., cnv_abc123) allows anyone with access to Front to navigate directly to the original conversation and verify the data independently. This is not a row number we assigned. It is the identifier from the system of record.

Why This Matters

These two requirements create a verifiable chain of custody. A skeptical reader can take any row in the dataset, read the evidence quote to understand what was observed, and then use the source system ID to locate the original record and confirm it. No row exists in isolation. Every data point is anchored to something real.

Data Hash

Each Proof Certificate includes a SHA-256 hash of the certified dataset. This hash is a cryptographic fingerprint: if even a single character in the dataset changes after certification, the hash will no longer match. This guarantees that the dataset you are reviewing is identical to the one that was verified. No rows have been added, removed, or modified after the fact.

2. Classification Verification

Classification Verification answers the question: Did the system interpret the data correctly?

Many deliverables involve classification ‚Äî assigning a category or label to each record. For example, a customer service message might be classified as "Damaged Product," "Wrong Item Shipped," or "Delivery Delay." Classification Verification confirms that these labels are supported by the evidence.

How It Works

Signal Detection. For each classification category, we define a set of expected signals ‚Äî specific words, phrases, or patterns that should appear in records assigned to that category. For example, a record classified as "Damaged Product" should contain signals such as "broken," "cracked," "damaged," "shattered," "dented," or descriptions of physical harm to the item. These signal lists are defined in advance, not generated on the fly.

Conflict Detection. We also define conflicting signals ‚Äî words or patterns that would contradict a given classification. For instance, a record classified as "Damaged Product" that contains the phrase "never received" suggests a delivery failure, not product damage. The presence of conflicting signals raises a flag.

Automated Comparison. Every row in the dataset is automatically checked against both its expected and conflicting signal lists. This produces one of three verdicts:

  • VERIFIED ‚Äî The evidence contains expected signals for the assigned classification and no conflicting signals. The label is well-supported.
  • LOW_CONFIDENCE ‚Äî The evidence does not contain strong expected signals, but no conflicting signals are present either. The label may be correct, but the evidence is ambiguous.
  • FLAGGED ‚Äî The evidence contains signals that conflict with the assigned classification. This row requires human review.

Plain English Explanations. Every verdict is accompanied by a human-readable explanation. Rather than presenting raw signal lists, the certificate explains in plain language why a row was verified, why confidence is low, or what conflict was detected. For example: "This row is classified as 'Wrong Item Shipped' and the evidence contains the phrase 'received a completely different product,' which supports this classification."

3. Integrity Scoring

Integrity Scoring answers the question: How trustworthy is the dataset as a whole?

Individual row-level verdicts are useful, but stakeholders also need a single summary measure. The Integrity Score combines three components into a letter grade from A+ through F.

The Three Components

Source Evidence Rate. The percentage of rows in the dataset that have both an evidence quote and a source system ID. A dataset where 95% of rows have complete source evidence scores 95% on this component. Missing evidence means a data point cannot be independently verified.

Classification Match Rate. The percentage of classified rows that received a VERIFIED verdict during signal detection. A high match rate means the assigned labels are well-supported by the evidence. This component only applies to datasets that include classifications.

Flag Rate. The percentage of rows that received a FLAGGED verdict ‚Äî meaning conflicting signals were detected. Unlike the other two components, a lower flag rate is better. A high flag rate indicates systematic misclassification or data quality problems.

How the Grade Is Calculated

The three components are combined using a weighted average:

  • Source Evidence Rate contributes 30% of the final score
  • Classification Match Rate contributes 50% of the final score
  • Flag Rate (inverted, so lower is better) contributes 20% of the final score

The resulting percentage maps to a letter grade. The full grading rubric is documented separately in the Confidence Grading reference.

Limitations

What this verification system can and cannot confirm

4. Limitations

No verification system is perfect. The Proof Certificate is explicit about what it can and cannot verify.

What This System Cannot Do

It cannot verify real-world outcomes. The certificate verifies that data was extracted correctly from source records and classified accurately. It does not verify what happened in the physical world. If a record says a replacement was shipped, the certificate confirms the record says that. It does not confirm the replacement arrived.

It cannot access external systems. Verification is limited to the data and source systems we have access to. If a record references a USPS tracking number, the certificate does not query USPS to confirm delivery status. If a record references an action taken in a system we cannot access, we can only verify what the record states, not the outcome.

Keyword detection is not natural language understanding. Signal detection works by identifying specific words and phrases. It does not understand context, sarcasm, negation patterns, or implied meaning the way a human reader does. A message that says "I wouldn't say it was damaged" contains the word "damaged" and might trigger a signal match even though the customer is saying the opposite. These edge cases are rare but real, which is why the system produces LOW_CONFIDENCE and FLAGGED verdicts rather than silently accepting everything.

Low confidence does not mean wrong. A LOW_CONFIDENCE verdict means the automated system could not find strong signal evidence to confirm the classification. It does not mean the classification is incorrect. Some records are accurately classified by a human analyst who understood nuance that keyword detection cannot capture. Low confidence is an invitation to review, not an accusation of error.

5. Anti-Hallucination Guarantees

The Proof Certificate system is designed from the ground up to prevent fabricated data ‚Äî what the AI industry calls "hallucination."

Every Data Point Is Extracted, Never Generated

Evidence quotes in the certificate are copied directly from source text. The system does not summarize, rephrase, or generate descriptions of what a source record "probably" says. If the exact text cannot be extracted, the evidence field is left empty and the row is marked accordingly. An empty evidence field is honest. A fabricated one is not.

Classifications Are Derived from Signals, Not Interpretation

When the system verifies a classification, it checks for the presence or absence of predefined signals. It does not ask "does this feel like a damaged product complaint?" It asks "does the text contain any of these 47 specific words and phrases associated with product damage?" This mechanical approach is less flexible than human judgment but far less susceptible to fabrication.

The Certificate Shows Raw Evidence for Independent Verification

The Proof Certificate is not a summary document that asks you to trust its conclusions. It shows the actual evidence ‚Äî the quotes, the source IDs, the signals detected, and the verdicts ‚Äî so that any reader can evaluate the conclusions independently. The certificate is a transparency tool, not a trust-me tool.

Data Hash Proves No Post-Hoc Modification

The SHA-256 hash included in every certificate is computed at the moment of certification. If the dataset is modified afterward ‚Äî even by a single character ‚Äî the hash will not match. This prevents a scenario where data is certified, then quietly edited before delivery. The hash is the seal. If the seal is broken, the certificate is void.

Summary

The Proof Certificate methodology rests on a simple principle: show your work. Every data point traces to a source. Every classification is checked against evidence. Every dataset receives an integrity grade. And every limitation is stated plainly. The result is a deliverable that does not require trust ‚Äî it provides proof.
